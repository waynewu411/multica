1. Overall Architecture

Multica is a monorepo combining a Go backend with a TypeScript frontend (pnpm workspaces + Turborepo). It also ships an Electron desktop app that shares nearly all code with the web app.

multica/
├── server/               # Go backend (Chi router, sqlc, gorilla/websocket, Redis)
├── apps/
│   ├── web/              # Next.js App Router (web app)
│   ├── desktop/          # Electron desktop app
│   └── docs/             # Documentation site
├── packages/
│   ├── core/             # Headless business logic (stores, API client, TanStack Query hooks)
│   ├── views/            # Shared page-level components (zero next/* imports)
│   ├── ui/               # Atomic UI component library (shadcn/ui + Base UI primitives)
│   └── tsconfig/         # Shared TypeScript configuration
├── e2e/                  # Playwright E2E tests
└── scripts/              # Development scripts

Key Architectural Decisions

  • Internal Packages pattern — shared packages export raw .ts/.tsx files (no pre-compilation). The consuming app's bundler compiles them directly, giving zero-config HMR and instant go-to-definition.
  • Dependency direction: views/ → core/ + ui/. Core and UI are independent of each other.
  • Platform bridge: packages/core/platform/CoreProvider initializes the API client, auth/workspace stores, WebSocket connection, and QueryClient. Each app wraps its root with <CoreProvider>.

2. Backend (Go)

The backend is a monolith built on Chi router with PostgreSQL (pgx driver) and gorilla/websocket.

Entry Point — server/cmd/server/main.go

On startup, the server:

  1. Connects to PostgreSQL via a pgxpool.Pool
  2. Creates the core trio: events.Bus (in-process pub/sub), realtime.Hub (WebSocket rooms), daemonws.Hub (daemon WS)
  3. Optionally creates Redis relay clients (when REDIS_URL is set) for multi-node fanout
  4. Constructs the Chi router, injecting all dependencies
  5. Registers event bus listeners: broadcast → activity → notification → subscriber → autopilot
  6. Starts background workers: runtime sweeper, heartbeat scheduler, autopilot scheduler, failure monitor
  7. Performs graceful shutdown via context cancellation

Route Structure — server/cmd/server/router.go

Handler Layer — server/internal/handler/

40+ handler files, all receiving a shared Handler struct that holds:

  • Queries — sqlc-generated type-safe DB access
  • Hub — realtime WebSocket hub for broadcasting events
  • Bus — event bus for internal pub/sub
  • TaskService — task lifecycle management
  • Storage — S3 or local file storage
  • Analytics — telemetry client

3. Event System

Event Bus — server/internal/events/bus.go

An in-process synchronous pub/sub bus. Handlers run in registration order; panics are recovered per-handler.

bus.Subscribe("issue:updated", handler)   // type-specific
bus.SubscribeAll(handler)                  // global (receives ALL events)
bus.Publish(Event{Type: "issue:updated", WorkspaceID: "...", Payload: {...}})

There are 70+ event types: issue:created, issue:updated, issue:deleted, comment:added, task:queued, task:dispatched, task:completed, task:failed, inbox:new, workspace:member_added, chat:message_added, autopilot:run_started, and many more.

Listener Chain (registered in order)

  1. Broadcaster Listener — routes events to the WebSocket layer. Workspace-scoped events broadcast to the workspace room; personal events (inbox, invitations) go via SendToUser to the specific recipient.
  2. Activity Listener — persists activity log entries to activity_log table for timeline display.
  3. Notification Listener — creates inbox items from events (mentions, status changes, assignments, comments). Respects notification preferences.

4. Real-Time System

WebSocket Hub — server/internal/realtime/hub.go

Scope-based room system. Clients connect via JWT or PAT and auto-subscribe to their workspace and user scopes. Additional scopes (task, chat) require authorization via ScopeAuthorizer.

ScopePatternUsage
Workspaceworkspace:{id}All workspace-level events
Useruser:{id}Personal events (inbox, invitations)
Tasktask:{id}Live task progress streaming
Chatchat:{id}Chat session messages
DaemonRuntimedaemon:runtime:{id}Daemon wakeup signals

Redis Relay (Multi-Node)

When REDIS_URL is set, the server enables cross-node fanout via Redis Streams:

  • ShardedStreamRelay — shards streams by workspace ID for horizontal scaling
  • DualWriteBroadcaster — writes to both the in-memory hub AND the Redis relay simultaneously
  • Supports legacy (single-stream), sharded, and dual (mirrored) modes for phased migration

Frontend WS Client — packages/core/api/ws-client.ts

Browser WebSocket wrapper with automatic reconnect (3-second delay) and event subscription via on(eventType, handler).

Frontend Realtime Sync — packages/core/realtime/use-realtime-sync.ts

Core pattern: WS events invalidate TanStack Query caches → refetch.

  • Debounced prefix-based invalidation (100ms) for rapid-fire events
  • Specific handlers for granular cache updates (issues, inbox, comments, chat, tasks)
  • Chat task lifecycle writethrough: task:queued / dispatch / message / completed / failed all update the chat pending-task cache inline

5. Daemon — Local Agent Runtime

This is Multica's core innovation. The daemon runs on a user's local machine and executes AI agent tasks.

Architecture — server/internal/daemon/daemon.go

  • Manages per-workspace runtime state (runtime IDs, repo URLs, settings)
  • Uses a taskRunner interface to execute agent CLIs (Claude Code, Codex, Cursor, Gemini Code Assist)
  • execenv/ package manages execution environment: git repo cloning, worktrees, sandbox isolation
  • repocache/ handles local git repo caching

Task Lifecycle

1
User creates an issue with an agent assigned
Handler inserts a task row into agent_task_queue
2
Server publishes task:queued event
TaskService notifies the daemon hub via NotifyTaskAvailable
3
Daemon receives daemon:task_available frame
Sent over the persistent daemon WebSocket connection
4
Daemon claims the task
POST /api/daemon/runtimes/{id}/tasks/claim
5
Server dispatches (task:dispatch event)
Daemon starts executing the agent locally in a sandboxed environment
6
Daemon streams progress
POST /api/daemon/tasks/{id}/progress and /messages
7
Task completes
POST /api/daemon/tasks/{id}/complete → task:completed event
8
Frontend shows the result in real-time
Chat or issue detail renders the agent's output

Daemon WebSocket — server/internal/daemonws/hub.go

Separate from user WebSocket connections, indexed by runtimeID. Handles heartbeats and task wakeup notifications. Includes deduplication (LRU seen-IDs) and slow-client eviction.

Reuses all shared packages. Additional concerns:

  • Drag region: macOS title bar drag strip on all full-window views

7. State Management

Architecture invariant: The system relies on a strict split between server state and client state. Mixing them is the most common way to break the architecture.
MechanismOwnsExamples
TanStack QueryAll server stateIssues, users, workspaces, inbox, agents — anything fetched from the API
ZustandAll client stateUI selections, filters, drafts, modal state, navigation history
React ContextCross-cutting platform plumbingWorkspaceIdProvider, NavigationProvider

Hard Rules

  • Never duplicate server data into Zustand. If it came from the API, it belongs in the Query cache.
  • Workspace-scoped queries must key on wsId. This makes workspace switching automatic — the cache key changes, the right data appears.
  • Mutations are optimistic by default. Apply locally, send request, roll back on failure, invalidate on settle.
  • WS events invalidate queries — they never write to stores directly. This keeps the cache as the single source of truth.
  • API responses must be parsed, not cast. Use parseWithFallback with Zod schemas to survive response drift (critical for the installed desktop app).

8. End-to-End Request Flow

Here is exactly how a user editing an issue flows through every layer:

1
React component calls a TanStack Query mutation
issues/mutations.ts → apiClient.updateIssue(id, data)
2
HTTP request sent
PUT /api/issues/{id} with Authorization: Bearer <token>, X-Workspace-Slug, CSRF token
3
Go middleware chain processes the request
RequestID → ClientMetadata → RequestLogger → CORS → Auth (validates JWT) → RequireWorkspaceMember (checks DB)
4
Handler validates input and updates PostgreSQL
sqlc-generated query writes to the issues table
5
Handler publishes issue:updated event to the Event Bus
Returns JSON response to the caller
6
Event Bus synchronously dispatches to listeners
a) Activity Listener — records row in activity_log
b) Notification Listener — creates inbox entries for subscribers
c) Broadcaster Listener — pushes to WebSocket hub
7
Broadcaster writes to Hub + Redis relay (multi-node)
DualWriteBroadcaster fans out to all connected clients in the workspace room
8
Browser WebSocket receives the message
onmessage fires → WSClient dispatches to registered handlers
9
useRealtimeSync updates the TanStack Query cache
Specific handler calls queryClient.setQueryData(); fallback invalidates by event prefix (100ms debounced)
10
React re-renders with updated data
Components watching the affected query keys see the new state
